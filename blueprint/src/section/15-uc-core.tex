The UC core is the reusable execution model of the project.  It is not the
EasyUC-paper interface: there are no hierarchical addresses
\(\alpha,\beta\), no input guard \(G\), and no interface constructor
\(\mathsf{MI}(F,A)\).  The current Lean model is built from natural-number
machine identities, explicit communication ports, message-level instructions,
proof-carrying execution setups, and the controller ensemble
\[
  \Ctrl.\leanid{exec}\;\sigma : \mathbb{N}\to\mathsf{PMF}\;\mathsf{Bool}.
\]
All definitions below are parametric in an arbitrary payload type \(\Pay\).

\subsection{Machines and Communication Sets}

\begin{definition}[Machine identities and ports]
\label{def:uc-machine-identities-ports}
\lean{LeanCryptoProtocols.UC.MachineId, LeanCryptoProtocols.UC.env_id, LeanCryptoProtocols.UC.adv_id, LeanCryptoProtocols.UC.PortLabel, LeanCryptoProtocols.UC.CommPort}
\leanok
Machine identities are natural numbers.  Two identities are reserved by the
core:
\[
  \Mid=\mathbb{N},\qquad \EnvId=0,\qquad \AdvId=1.
\]
A communication port is a triple
\[
  p=(p.\leanid{owner},p.\leanid{dest},p.\leanid{label}) : \Port
\]
with label
\[
  p.\leanid{label}\in
  \{\leanid{input},\leanid{subroutineOutput},\leanid{backdoor}\}.
\]
The port carries a proof of well-formedness:
\[
\begin{aligned}
  p.\leanid{owner} &\ne p.\leanid{dest},\\
  p.\leanid{label}=\leanid{backdoor}
    &\Longleftrightarrow
      (p.\leanid{owner}=\AdvId\lor p.\leanid{dest}=\AdvId).
\end{aligned}
\]
Thus adversarial communication is represented by ordinary routed messages on
backdoor-labelled ports.  There is no separate adversarial delivery channel.
\end{definition}

\begin{definition}[Messages and envelopes]
\label{def:uc-messages-envelopes}
\lean{LeanCryptoProtocols.UC.MessageInstruction, LeanCryptoProtocols.UC.Message, LeanCryptoProtocols.UC.Message.corrupt_instruction_implies_backdoor, LeanCryptoProtocols.UC.Envelope}
\leanok
A message stores a claimed source, a port label, a core instruction, and a
payload:
\[
  m=(m.\leanid{source},m.\leanid{label},
      m.\leanid{instruction},m.\leanid{payload}) : \Msg\;\Pay .
\]
The instruction is either ordinary traffic or a static-corruption request:
\[
  m.\leanid{instruction}\in
  \{\leanid{plain},\leanid{corrupt}(pid)\}.
\]
The instruction is not a port label and it does not choose the envelope
destination.  The core invariant is
\[
  m.\leanid{instruction}=\leanid{corrupt}(pid)
  \Longrightarrow
  m.\leanid{label}=\leanid{backdoor}.
\]
An envelope adds the destination port and proves that the port label and the
message label agree:
\[
  e=(e.\leanid{port},e.\leanid{message}) : \Envlp\;\Pay,
  \qquad e.\leanid{port.label}=e.\leanid{message.label}.
\]
The destination identity is read from the port.  The source identity is stored
inside the message; it is not trusted by itself, although honest machines
normally set it to their own identity when the protocol requires that
convention.
\end{definition}

\begin{definition}[Machine programs]
\label{def:uc-machine-programs}
\lean{LeanCryptoProtocols.UC.ActivationResult, LeanCryptoProtocols.UC.MachineProgram, LeanCryptoProtocols.UC.Machine, LeanCryptoProtocols.UC.AnyMachine, LeanCryptoProtocols.UC.AnyMachineState}
\leanok
A machine is a proof-carrying identity, static communication set, and local
program:
\[
  \mu=(\mu.\leanid{id},\mu.\leanid{communication\_set},
      \mu.\leanid{program}) : \leanid{Machine}\;\Pay\;O .
\]
The local program provides an initial state, a receive transition, a resume
distribution, a halt predicate, and an output function:
\[
\begin{array}{rcl}
\leanid{init} &:& \mathbb{N}\to \leanid{LocalState},\\
\leanid{receive} &:&
  \leanid{LocalState}\to \Msg\;\Pay\to \leanid{LocalState},\\
\leanid{resume} &:&
  \leanid{LocalState}\to
    \mathsf{PMF}(\leanid{ActivationResult}\;\Pay\;\leanid{LocalState}),\\
\leanid{is\_halted} &:& \leanid{LocalState}\to\mathsf{Bool},\\
\leanid{output} &:& \leanid{LocalState}\to O .
\end{array}
\]
One activation emits at most one envelope.  The machine well-formedness proof
requires every static port to be owned by \(\mu.\leanid{id}\), and requires at
most one static port to each destination identity.  Heterogeneous machines and
their local states are erased to \(\leanid{AnyMachine}\) and
\(\leanid{AnyMachineState}\) when they are stored in a protocol.
\end{definition}

\subsection{Protocols}

\begin{definition}[Protocol shape]
\label{def:uc-protocol-shape}
\lean{LeanCryptoProtocols.UC.Protocol, LeanCryptoProtocols.UC.machine_ids, LeanCryptoProtocols.UC.default_machine_states}
\leanok
A protocol is a list of machines plus proof obligations:
\[
  \pi : \leanid{Protocol}\;\Pay .
\]
Its computational fields are
\[
\begin{array}{ll}
\pi.\leanid{machines}
  &: \mathsf{List}(\leanid{AnyMachine}\;\Pay),\\
\pi.\leanid{initial\_states}
  &: \mathbb{N}\to
     \mathsf{PMF}(\mathsf{List}(\leanid{AnyMachineState}\;\Pay)).
\end{array}
\]
The default initial-state sampler initializes each listed machine with its own
\(\leanid{init}\) function.  The proof fields assert that machine identities
are unique, \(\EnvId\) and \(\AdvId\) are not protocol-machine identities,
protocol machines never send directly to \(\EnvId\), and any static port whose
destination is \(\AdvId\) is a backdoor port.

Current Lean code does not yet include a protocol-level
\(\leanid{corruptible\_machines}\) field.  The static corruption pattern used
by a security theorem is stored at the execution-setup layer, described in
Definition~\ref{def:uc-execution-setup}.
\end{definition}

\begin{definition}[Call graph and external identities]
\label{def:uc-call-graph}
\lean{LeanCryptoProtocols.UC.can_send_input_to, LeanCryptoProtocols.UC.can_send_subroutine_output_to, LeanCryptoProtocols.UC.is_caller_of_id, LeanCryptoProtocols.UC.is_subroutine_of_id, LeanCryptoProtocols.UC.Protocol.is_main_machine, LeanCryptoProtocols.UC.Protocol.is_internal_machine, LeanCryptoProtocols.UC.Protocol.is_external_identity_of}
\leanok
The caller and subroutine relations are derived from ports:
\[
\begin{array}{rcl}
\leanid{is\_caller\_of\_id}\;\mu\;j
  &\Longleftrightarrow&
  \mu \hbox{ has an input port to } j,\\
\leanid{is\_subroutine\_of\_id}\;\mu\;j
  &\Longleftrightarrow&
  \mu \hbox{ has a subroutine-output port to } j .
\end{array}
\]
The protocol proof fields require these two relations to match on internal
protocol calls.  A machine is a main machine when it has an external identity:
an identity that is not itself a protocol machine, but that is connected to the
machine by a subroutine-output port.  The predicates
\(\pi.\leanid{is\_main\_machine}\),
\(\pi.\leanid{is\_internal\_machine}\), and
\(\pi.\leanid{is\_external\_identity\_of}\) are the controller-facing
abstractions of this call graph.
\end{definition}

\begin{definition}[Environment, adversary, simulator, and setup]
\label{def:uc-execution-setup}
\lean{LeanCryptoProtocols.UC.Environment, LeanCryptoProtocols.UC.Adversary, LeanCryptoProtocols.UC.Simulator, LeanCryptoProtocols.UC.ExecutionSetup}
\leanok
The environment and adversary are machines with fixed identities:
\[
  E : \leanid{Environment}\;\Pay,\qquad
  A : \leanid{Adversary}\;\Pay .
\]
The environment has identity \(\EnvId\) and a unique backdoor port to
\(\AdvId\).  The adversary has identity \(\AdvId\), a unique backdoor port to
\(\EnvId\), and no field recording corrupted parties.  In the ideal world a
simulator is definitionally an adversary machine:
\[
  \leanid{Simulator}\;\Pay := \leanid{Adversary}\;\Pay .
\]
For a concrete triple \((\pi,A,E)\), an execution is admitted through a setup
witness
\[
  \sigma : \Setup\;\pi\;A\;E .
\]
The setup records the static corruption pattern:
\[
  \sigma.\leanid{corrupted\_parties} : \mathsf{Finset}\;\Mid .
\]
It also proves the execution-level port policies: environment ports either
target the adversary backdoor or input a main protocol machine, and static
adversary ports target only the environment.  Adversary-to-protocol control
ports are not stored in the adversary record; they are added by the runtime
overlay in Definition~\ref{def:uc-runtime-communication}.  Since the current
code has no protocol-level corruptibility universe, membership in
\(\sigma.\leanid{corrupted\_parties}\) is the controller's only core-level
authorization test for a corruption instruction.
\end{definition}

\subsection{Ideal Functionalities and Ideal Protocols}

\begin{definition}[Ideal functionality]
\label{def:uc-ideal-functionality}
\lean{LeanCryptoProtocols.UC.IdealFunctionality}
\leanok
An ideal functionality is a record.  We write
\[
  f:\IdealFunc\;\Pay .
\]
Besides the functionality machine, it stores the party identities and the
interface data needed to generate dummy parties:
\[
\begin{array}{ll}
f.\leanid{party\_ids}
  &: \mathsf{List}\;\Mid,\\
f.\leanid{functionality\_id}
  &: \Mid,\\
f.\leanid{machine}
  &: \leanid{Machine}\;\Pay\;\mathsf{Unit},\\
f.\leanid{party\_external\_ids}
  &: \Mid\to\mathsf{Finset}\;\Mid,\\
f.\leanid{wrap\_input}
  &: \mathsf{Option}\;\Mid\to\Pay\to\Pay,\\
f.\leanid{unwrap\_output}
  &: \Pay\to\mathsf{Option}(\Mid\times\Pay).
\end{array}
\]
The proof fields separate parties, external identities, the functionality,
\(\EnvId\), and \(\AdvId\).  They also constrain the functionality machine's
communication set: it may send subroutine outputs to parties and backdoor
messages to the adversary, but not arbitrary traffic.
\end{definition}

\begin{definition}[Dummy parties]
\label{def:uc-dummy-parties}
\lean{LeanCryptoProtocols.UC.DummyParty, LeanCryptoProtocols.UC.DummyParty.to_any_machine, LeanCryptoProtocols.UC.mk_dummy_party, LeanCryptoProtocols.UC.mk_dummy_parties}
\leanok
For each \(pid\in f.\leanid{party\_ids}\), the core constructs
\[
  \leanid{mk\_dummy\_party}\;f\;pid : \DummyParty\;\Pay .
\]
The generated dummy party has one input port to
\(f.\leanid{functionality\_id}\), subroutine-output ports to the external
identities \(f.\leanid{party\_external\_ids}\;pid\), and no backdoor port.  Its
resume behavior is fixed:
\begin{itemize}
\item A pending input is wrapped as
\[
  f.\leanid{wrap\_input}\;m.\leanid{source}\;m.\leanid{payload}
\]
and sent to \(f\).
\item A pending subroutine output is parsed by
\[
  f.\leanid{unwrap\_output}\;m.\leanid{payload}.
\]
If parsing succeeds and the destination is one of the dummy party's external
identities, the parsed payload is forwarded to that destination.
\item A pending backdoor message produces no outgoing envelope.
\end{itemize}
Thus dummy parties are generated once from the ideal functionality interface.
The simulator does not directly control dummy parties.  Any corrupted-party
leakage or command semantics in the ideal world must be implemented by the
ideal functionality program and its backdoor/control interface.
\end{definition}

\begin{definition}[Ideal protocol generated from a functionality]
\label{def:uc-ideal-protocol}
\lean{LeanCryptoProtocols.UC.IdealProtocol, LeanCryptoProtocols.UC.mk_ideal_protocol, LeanCryptoProtocols.UC.mk_ideal_protocol_machine_ids, LeanCryptoProtocols.UC.mk_ideal_protocol_is_main_machine_iff, LeanCryptoProtocols.UC.mk_ideal_protocol_is_internal_machine_iff}
\leanok
The ideal protocol is produced by
\[
  \leanid{mk\_ideal\_protocol}\;f : \IdealProtocol\;\Pay .
\]
Its protocol machine list is
\[
  \leanid{mk\_dummy\_parties}\;f
  \quad\hbox{followed by}\quad f.\leanid{machine}.
\]
The constructor proves the ordinary protocol-shape obligations.  The generated
machine identities are exactly
\[
  f.\leanid{party\_ids} \mathbin{++} [f.\leanid{functionality\_id}],
\]
the generated main machines are exactly the dummy parties, and the unique
internal machine is the ideal functionality.  There is no global adversary
visibility set in this construction.  Simulator interaction with the ideal
world is ordinary backdoor/control communication with the functionality; the
functionality program decides what those payloads mean.
\end{definition}

\subsection{Controller Execution}

\begin{definition}[Runtime communication set]
\label{def:uc-runtime-communication}
\lean{LeanCryptoProtocols.UC.ExecutionSetup.protocol_comm_set, LeanCryptoProtocols.UC.ExecutionSetup.backdoor_port_to_adversary, LeanCryptoProtocols.UC.ExecutionSetup.backdoor_port_from_adversary, LeanCryptoProtocols.UC.ExecutionSetup.env_input_port_to_main, LeanCryptoProtocols.UC.ExecutionSetup.runtime_communication_set}
\leanok
The controller does not route messages solely through a machine's static
communication set.  For a setup \(\sigma\) and active sender \(i\), it computes
\[
  \sigma.\leanid{runtime\_communication\_set}\;i
  : \mathsf{Finset}\;\Port .
\]
This set is the union of the sender's base ports and a runtime overlay:
\[
\begin{array}{ll}
i=\EnvId
  &: \hbox{environment ports plus inputs to all main machines},\\
i=\AdvId
  &: \hbox{adversary ports plus backdoors to all protocol machines},\\
i\notin\{\EnvId,\AdvId\}
  &: \hbox{protocol ports plus a backdoor to the adversary}.
\end{array}
\]
This overlay gives control-communication routes, not corruption authority.
Ordinary \(\leanid{plain}\) backdoor messages are routed if the envelope is
otherwise valid, and the target machine program decides whether the payload has
any effect.
\end{definition}

\begin{definition}[Outgoing-message validity]
\label{def:uc-outgoing-validity}
\lean{LeanCryptoProtocols.UC.ExecutionSetup.environment_source_valid, LeanCryptoProtocols.UC.ExecutionSetup.sender_source_valid, LeanCryptoProtocols.UC.ExecutionSetup.corruption_instruction_valid, LeanCryptoProtocols.UC.ExecutionSetup.outgoing_message_valid}
\leanok
Before routing an emitted envelope, the controller checks
\[
  \sigma.\leanid{outgoing\_message\_valid}\;i\;e .
\]
This predicate requires the envelope's port to be owned by the active sender
and to occur in the sender's runtime communication set:
\[
  e.\leanid{port.owner}=i,\qquad
  e.\leanid{port}\in
    \sigma.\leanid{runtime\_communication\_set}\;i .
\]
It also checks the source convention.  If \(i=\EnvId\), a message sent to
\(\AdvId\) must have source \(\leanid{none}\), while a message sent to a
protocol machine must claim one of that machine's external identities.  If
\(i\ne\EnvId\), the source must be \(\leanid{some}\;i\).

The corruption-instruction check is exactly
\[
\begin{array}{rcl}
\leanid{plain}
  &\Longmapsto& \mathsf{True},\\
\leanid{corrupt}(pid)
  &\Longmapsto&
    e.\leanid{message.label}=\leanid{backdoor}
    \land pid\in\sigma.\leanid{corrupted\_parties}.
\end{array}
\]
The corrupted identity is read only from the instruction.  It is not inferred
from the sender, receiver, or claimed source.  In particular, an ideal-world
simulator may send a backdoor envelope to an ideal functionality with
\(\leanid{instruction}=\leanid{corrupt}(pid)\); the controller checks only
whether \(pid\) is in the setup's static corruption pattern, and the
functionality program defines the resulting leakage or command semantics.
\end{definition}

\begin{definition}[Controller state and step]
\label{def:uc-controller-step}
\lean{LeanCryptoProtocols.UC.ProtocolMachineState, LeanCryptoProtocols.UC.ControllerState, LeanCryptoProtocols.UC.Controller.initial_state_with_protocol_states, LeanCryptoProtocols.UC.Controller.route_envelope, LeanCryptoProtocols.UC.Controller.handle_outgoing, LeanCryptoProtocols.UC.Controller.step}
\leanok
A controller state stores the local state of \(E\), the local state of \(A\),
the list of protocol-machine states, and the active identity:
\[
  s=(s_E,s_A,s_\pi,i_{\mathsf{active}})
    : \leanid{ControllerState}\;\sigma .
\]
The initial active identity is \(\EnvId\).  One step first checks whether the
environment has halted.  If not, it resumes the active machine:
\[
\begin{array}{c|c}
i_{\mathsf{active}} & \hbox{resumed component}\\ \hline
\EnvId & E,\\
\AdvId & A,\\
\hbox{protocol identity} & \hbox{the matching protocol machine}.
\end{array}
\]
The optional outgoing envelope is validated by
Definition~\ref{def:uc-outgoing-validity}.  Valid envelopes are routed to the
destination machine, adversary, or environment.  Invalid envelopes are dropped
and control returns to the environment.
\end{definition}

\begin{definition}[Execution ensemble]
\label{def:uc-execution-ensemble}
\lean{LeanCryptoProtocols.UC.Controller.run_steps, LeanCryptoProtocols.UC.Controller.run_steps_with_status, LeanCryptoProtocols.UC.Controller.exec, LeanCryptoProtocols.UC.Controller.exec_with_status, LeanCryptoProtocols.UC.Controller.run_steps_output_eq_of_step_map}
\leanok
The controller runs for the fixed project step budget
\(\leanid{LeanCryptoProtocols.max\_controller\_steps}\), stopping earlier if
the environment halts.  The security parameter \(n\) indexes machine
initialization and protocol initial-state sampling; it is not the step budget.
For a setup \(\sigma\), the execution ensemble is
\[
  \Run(\sigma)=\Ctrl.\leanid{exec}\;\sigma
  : \mathbb{N}\to\mathsf{PMF}\;\mathsf{Bool}.
\]
The observed bit is the final environment output:
\[
  \Pr[\Run(\sigma,n)=1]
  =\ProbOne(\Ctrl.\leanid{exec}\;\sigma\;n).
\]
The generic controller lifting lemma
\(\leanid{run\_steps\_output\_eq\_of\_step\_map}\) is the intended way to turn
step-level path matching into equality of controller output distributions.
\end{definition}

\subsection{UC Security Definitions}

\begin{definition}[Indistinguishability primitives]
\label{def:uc-indistinguishability-primitives}
\lean{LeanCryptoProtocols.UC.Ensemble, LeanCryptoProtocols.UC.PPT, LeanCryptoProtocols.UC.Negligible, LeanCryptoProtocols.UC.SecurityLevel, LeanCryptoProtocols.UC.probTrue, LeanCryptoProtocols.UC.Distinguisher, LeanCryptoProtocols.UC.ComputationalIndist}
\leanok
An ensemble is a security-parameter-indexed family of distributions:
\[
  X : \mathbb{N}\to\mathsf{PMF}\;\alpha .
\]
The project uses three security levels:
\[
  \SecLevel=\{
    \leanid{perfect},\leanid{statistical},\leanid{computational}
  \}.
\]
Perfect indistinguishability is pointwise equality of ensembles.  Statistical
indistinguishability uses a negligible bound on total variation distance.
Computational indistinguishability quantifies over all polynomial-time
distinguishers
\[
  D : \leanid{Distinguisher}\;\alpha,\qquad
  D.\leanid{run}:\alpha\to\mathbb{N}\to\mathsf{PMF}\;\mathsf{Bool}.
\]
The predicate \(\PPTM\) is an abstract complexity interface shared by
machines, distinguishers, and reductions.
\end{definition}

\begin{definition}[Execution difference]
\label{def:uc-exec-diff}
\lean{LeanCryptoProtocols.UC.exec_diff}
\leanok
For real and ideal setups over the same environment, Lean defines
\[
\begin{aligned}
  \ExecDiff(\sigma_r,\sigma_i,n)
    =
      \left|
        \ProbOne(\Ctrl.\leanid{exec}\;\sigma_r\;n)
        -
        \ProbOne(\Ctrl.\leanid{exec}\;\sigma_i\;n)
      \right|.
\end{aligned}
\]
Perfect UC security is stated as equality of the full
\(\mathsf{PMF}\;\mathsf{Bool}\).  Statistical and computational UC security
use \(\ExecDiff\) with a negligible upper bound.
\end{definition}

\begin{definition}[UC-emulation at a static corruption pattern]
\label{def:uc-emulation}
\lean{LeanCryptoProtocols.UC.UCEmulatesAt, LeanCryptoProtocols.UC.UCEmulatesPerfect, LeanCryptoProtocols.UC.UCEmulatesPerfectWithCorruption, LeanCryptoProtocols.UC.UCEmulatesStatistical, LeanCryptoProtocols.UC.UCEmulatesStatisticalWithCorruption, LeanCryptoProtocols.UC.UCEmulatesComputational, LeanCryptoProtocols.UC.UCEmulatesComputationalWithCorruption}
\leanok
For protocols \(\pi\) and \(\varphi\), Lean defines
\[
  \UCEmulates(\ell,\Corr,\pi,\varphi)
\]
by cases on \(\ell:\SecLevel\).  The finite set \(\Corr\) is the static
corruption pattern being compared.  It is a theorem parameter, not a field of
the adversary or simulator.

In the computational branch the shape is
\[
\begin{aligned}
&\forall A,\ \PPTM(A)\Rightarrow
  \exists S,\ \PPTM(S)\land\\
&\quad
  \forall E,\ \PPTM(E)\Rightarrow\\
&\quad
  \forall\sigma_r:\Setup\;\pi\;A\;E,\;
  \forall\sigma_i:\Setup\;\varphi\;S\;E,\\
&\quad
  \sigma_r.\leanid{corrupted\_parties}=\Corr
  \Rightarrow
  \sigma_i.\leanid{corrupted\_parties}=\Corr
  \Rightarrow\\
&\quad
  \exists\epsilon,\ \Negl(\epsilon)\land
  \forall n,\ \ExecDiff(\sigma_r,\sigma_i,n)\le\epsilon(n).
\end{aligned}
\]
The perfect branch replaces the negligible bound by equality of controller
output distributions for all \(n\).  The statistical branch omits the
\(\PPTM\) premises but keeps the negligible \(\ExecDiff\) bound.  The aliases
without \(\leanid{WithCorruption}\) instantiate \(\Corr=\varnothing\); the
\(\leanid{WithCorruption}\) aliases keep \(\Corr\) explicit.
\end{definition}

\begin{definition}[UC-realization at a static corruption pattern]
\label{def:uc-realization}
\lean{LeanCryptoProtocols.UC.UCRealizesAt, LeanCryptoProtocols.UC.UCRealizesPerfect, LeanCryptoProtocols.UC.UCRealizesPerfectWithCorruption, LeanCryptoProtocols.UC.UCRealizesStatistical, LeanCryptoProtocols.UC.UCRealizesStatisticalWithCorruption, LeanCryptoProtocols.UC.UCRealizesComputational, LeanCryptoProtocols.UC.UCRealizesComputationalWithCorruption}
\leanok
UC-realization is UC-emulation against the ideal protocol generated from an
ideal functionality:
\[
  \UCRealizes(\ell,\Corr,\pi,f)
  \quad\hbox{uses}\quad
  \varphi_f=(\leanid{mk\_ideal\_protocol}\;f).\leanid{protocol}.
\]
There is no additional global visibility premise.  If the ideal world needs a
simulator interface, the ideal functionality machine must expose the relevant
backdoor/control behavior through its own communication set and program.  The
same finite set \(\Corr\) is required of the real and ideal setups:
\[
  \sigma_r.\leanid{corrupted\_parties}=\Corr,\qquad
  \sigma_i.\leanid{corrupted\_parties}=\Corr .
\]
Again, the no-corruption aliases instantiate \(\Corr=\varnothing\), while the
\(\leanid{WithCorruption}\) aliases keep the static pattern explicit.
\end{definition}
